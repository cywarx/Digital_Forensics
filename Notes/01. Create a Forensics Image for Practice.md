
> [!tip] Goal
> Create a 25 MB virtual pendrive image for forensics practice using `dd` and `dc3dd`.

---

## Step 1 — Create the Image

```bash
dd if=/dev/zero of=pendrive.img bs=1M count=25
```

> [!info]- How does this make 25 MB?
> - `bs=1M` → each block = **1 MB**
> - `count=25` → write **25 blocks**
> - `1 MB × 25 = 25 MB` total image

---

## Step 2 — Hash with dc3dd

```bash
dc3dd if=pendrive.img hash=sha256 log=pendrive.log
```

> [!info]- Why hash?
> Generates a **SHA-256** cryptographic hash to prove the image has not been tampered with — essential for forensic integrity.
> - `hash=sha256` → hashing algorithm
> - `log=pendrive.log` → saves hash to a log file

---

## Step 3 — Format as FAT32

```bash
mkfs.vfat pendrive.img
```

> [!info]- Why FAT32?
> Adds a filesystem so the image behaves like a real pendrive. FAT32 is the most common format for USB drives.

> [!info]- Why does pendrive_mount show Jan 1 1970 after formatting?
> FAT32 filesystem does not store a creation date for the root directory — it defaults to **Unix Epoch (Jan 1 1970)** which means `0` in Unix time, i.e. "no date recorded". This is completely normal.
>

> |--|--|
> | FAT32 | ❌ No — defaults to Jan 1 1970 |
> | ext4 | ✅ Yes — shows 
## Step 4 — Mount

```bash
mkdir ~/Desktop/Forensics_Image/pendrive_mount
sudo mount -o loop pendrive.img ~/Desktop/Forensics_Image/pendrive_mount
```

> [!warning]- RO Flag — Read-Only vs Read-Write
> Check `lsblk` after mounting:
> - `RO=1` → **Read-Only** — you cannot write anything, even with `sudo`
> - `RO=0` → **Read-Write** — you can add files via the mount point
>
> If mounted read-only, force read-write:
> ```bash
> sudo mount -o loop,rw pendrive.img ~/Desktop/Forensics_Image/pendrive_mount
> ```

> [!warning]- Why does sudo change ownership of pendrive_mount?
> Mounting with `sudo` makes **root** the owner of the mount point — so normal users get `Permission denied` when creating files.
>
> **Before mount:**
> ```zsh
> ┌──(HeXx㉿UbuntuVM)-[~/Desktop/Forensics_Image]
> └─$ ls -l
> total 25608
> -rw-rw-r-- 1 warrior warrior 26214400 Mar 28 11:43 pendrive.img
> -rw-rw-r-- 1 warrior warrior      470 Mar 28 11:40 pendrive.log
> drwxrwxr-x 2 warrior warrior     4096 Mar 28 11:47 pendrive_mount
> ```
>
> **After mount with sudo:**
> ```zsh
> ┌──(HeXx㉿UbuntuVM)-[~/Desktop/Forensics_Image]
> └─$ ls -l
> total 25620
> -rw-rw-r-- 1 warrior warrior 26214400 Mar 28 11:48 pendrive.img
> -rw-rw-r-- 1 warrior warrior      470 Mar 28 11:40 pendrive.log
> drwxr-xr-x 2 root    root       16384 Jan  1  1970 pendrive_mount
> ```
>
> Notice: owner changed from `warrior` to `root`, date changed to `Jan 1 1970`, size changed to `16384`.
>
> Fix by mounting with your own uid/gid:
> ```bash
> sudo mount -o loop,uid=1000,gid=1000 pendrive.img ~/Desktop/Forensics_Image/pendrive_mount
> ```
> Now you can create files without `sudo`. Check your uid/gid with `id`.

> [!info]- Why create a mount directory?
> `pendrive.img` is the **actual storage** — `pendrive_mount/` is just the **window to access it**.
>
> ```
> pendrive.img     ← actual disk (locked box)
> pendrive_mount/  ← window/door to open it
> ```
>
> Always add files inside `pendrive_mount/` — they automatically get saved into `pendrive.img`.
>
> | | `pendrive.img` | `pendrive_mount/` |
> |--|--|--|
> | What is it | The actual disk | The window/door |
> | Add files here? | ❌ No | ✅ Yes |
> | Files saved to | — | `pendrive.img` |
>
> > Think of `pendrive.img` as a **locked box** and `pendrive_mount/` as the **key that opens it**. You always interact through the key, not the box directly.

> [!info]- What is `-o loop`?
> A **loop device** lets you mount a regular file as if it were a physical disk.
>
> **Normal disk:**
> ```
> Physical USB → /dev/sdb → mount → access files
> ```
>
> **Loop device:**
> ```
> pendrive.img → /dev/loop0 → mount → access files
> ```
>
> Linux creates a virtual device (like `/dev/loop0`) pointing to your `pendrive.img`, tricking the system into treating it like a real drive.
>
> `-o loop` = *"treat this file as a disk"*

> [!info]- Leftover loop devices — how to remove
> Sometimes old loop devices stay attached even after unmounting. Check with:
> ```bash
> sudo losetup -l
> ```
> Detach them:
> ```bash
> sudo losetup -d /dev/loop12
> ```
> If still showing in file explorer, refresh Nautilus:
> ```bash
> nautilus -q
> ```


> [!tip]- Practice Tip

## Step 6 — Unmount

```bash
sudo umount ~/Desktop/Forensics_Image/pendrive_mount
```

> [!warning]- Important
> Always unmount before analyzing the image with forensic tools like Autopsy or Sleuth Kit.
