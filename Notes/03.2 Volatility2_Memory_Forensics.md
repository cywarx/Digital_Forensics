## вљ™пёЏ Phase 0 вЂ” Environment Setup

> [!tip] Before You Start
> Set a shell variable for the dump path to avoid repetition. All commands below use `$DUMP` and `$PROFILE`.

```bash
# Set your dump path once
export DUMP="/path/to/memory.raw"
export PROFILE="Win7SP1x64"   # fill in after Phase 1

# Verify Volatility2 is working
vol.py --info | head -30
vol.py --info | grep -i "Win7"   # list available Win7 profiles
```

> [!note] Plugin Path (if using custom plugins like vol2-community)
> ```bash
> vol.py --plugins=/opt/volatility-plugins/ -f $DUMP --profile=$PROFILE <plugin>
> ```

## рџ”Ќ Phase 1 вЂ” Profile Identification (ALWAYS FIRST)

> [!warning] Never skip this phase
> Running any plugin with the **wrong profile** gives garbage output.  
> Always identify the OS/arch before anything else.

### Step 1.1 вЂ” `imageinfo` (Primary Method)

```bash
vol.py -f $DUMP imageinfo
```

**What it does:** Scans the dump for KDBG signatures, guesses the best matching profile(s), and returns OS version, architecture, build number, and suggested profiles.

**What to look for:**
- `Suggested Profile(s)` вЂ” pick the first one (most likely match)
- `Image date and time` вЂ” system time at capture
- `Number of processors`

**Example output:**
```
Suggested Profile(s) : Win7SP1x64, Win7SP0x64, Win2008R2SP0x64
AS Layer1 : WindowsAMD64PagedMemory ...
Image date and time  : 2024-03-15 14:22:11 UTC+0000
```

---

### Step 1.2 вЂ” `kdbgscan` (Fallback / Verification)

```bash
vol.py -f $DUMP kdbgscan
```

**What it does:** Searches for all KDBG (Kernel Debugger Block) structures in memory. More thorough than `imageinfo`. Use when `imageinfo` returns multiple conflicting profiles or when dealing with hibernation files.

**What to look for:**
- Multiple KDBG hits вЂ” pick the one with the highest `offset` (usually the most recent/active)
- `Build string` confirms OS version directly from kernel

> [!tip] Once profile is confirmed
> ```bash
> export PROFILE="Win7SP1x64"
> ```

## рџ§© Phase 2 вЂ” Process Analysis

> [!info] Goal
> Build a **complete process picture** вЂ” what was running, in what order, what was hidden, and what was suspicious.

### Step 2.1 вЂ” `pslist` (Active Process List from EPROCESS)

```bash
vol.py -f $DUMP --profile=$PROFILE pslist
```

**What it does:** Walks the `PsActiveProcessHead` doubly-linked list in the kernel to enumerate all active processes at capture time. Shows PID, PPID, name, start time, exit time, threads, handles.

**What to look for:**
- Processes with no parent (PPID = 0 or PPID doesn't exist)
- Suspicious process names (misspellings: `svch0st.exe`, `lsas.exe`)
- Processes running from `%TEMP%`, `%APPDATA%`, or unusual paths
- Orphaned processes (parent already exited but child still running)
- Duplicate `svchost.exe` entries вЂ” normal, but check their PPIDs (should all be `services.exe`)

---

### Step 2.2 вЂ” `pstree` (Visual Process Hierarchy)

```bash
vol.py -f $DUMP --profile=$PROFILE pstree
```

**What it does:** Same as `pslist` but renders the parent-child relationships as an indented tree. Makes spotting anomalous parent-child pairs trivial.

**What to look for (known-bad parent-child combos):**
- `cmd.exe` spawned by `word.exe`, `excel.exe`, `acrobat.exe` в†’ macro/exploit
- `powershell.exe` spawned by `svchost.exe` в†’ LOLBin abuse
- `mshta.exe`, `wscript.exe`, `cscript.exe` spawned by browser/doc в†’ dropper
- `cmd.exe` or `powershell.exe` spawned by a random PID that no longer exists

---

### Step 2.3 вЂ” `psscan` (Scan for Hidden/Terminated Processes)

```bash
vol.py -f $DUMP --profile=$PROFILE psscan
```

**What it does:** Instead of walking the linked list (which rootkits can unlink themselves from), it **pool-scans** the entire memory for `EPROCESS` pool tags. Finds processes that:
- Were unlinked from `pslist` (DKOM вЂ” Direct Kernel Object Manipulation)
- Have already **terminated** but whose memory hasn't been freed yet

> [!important] Cross-reference `pslist` vs `psscan`
> ```bash
> # PIDs in psscan but NOT in pslist = potentially hidden or terminated
> vol.py -f $DUMP --profile=$PROFILE pslist | awk '{print $3}' > pslist_pids.txt
> vol.py -f $DUMP --profile=$PROFILE psscan | awk '{print $3}' > psscan_pids.txt
> diff pslist_pids.txt psscan_pids.txt
> ```

---

### Step 2.4 вЂ” `psxview` (Multi-Source Process Comparison)

```bash
vol.py -f $DUMP --profile=$PROFILE psxview
```

**What it does:** The ultimate process hiding detector. Checks **7 different sources** simultaneously and marks `True/False` for each:
- `pslist` (EPROCESS linked list)
- `psscan` (pool scan)
- `thrdproc` (via thread ETHREAD objects)
- `pspcdh` (PspCidTable handle table)
- `csrss` (CSRSS handle table)
- `session` (session space objects)
- `deskthrd` (desktop thread objects)

**What to look for:**
- Any row where columns disagree (e.g., `psscan = True` but `pslist = False`) = **process hiding detected**

---

### Step 2.5 вЂ” `cmdline` (Command Line Arguments per Process)

```bash
vol.py -f $DUMP --profile=$PROFILE cmdline
```

**What it does:** Reads the `PEB.ProcessParameters.CommandLine` field for every process to show the **exact command-line string** used to launch it.

**What to look for:**
- `powershell.exe` with `-enc`, `-nop`, `-w hidden`, `-exec bypass` в†’ encoded/obfuscated commands
- `cmd.exe /c` executing base64 strings or temp-path binaries
- Suspicious arguments like IP addresses, ports, URLs in process cmdlines
- `schtasks`, `reg`, `net`, `wmic` being run inline

---

### Step 2.6 вЂ” `dlllist` (DLLs Loaded per Process)

```bash
# All processes
vol.py -f $DUMP --profile=$PROFILE dlllist

# Specific process by PID
vol.py -f $DUMP --profile=$PROFILE dlllist -p 1234
```

**What it does:** Reads the `PEB.Ldr` (Loader Data) structure to list every DLL mapped into a process, with full path and load address.

**What to look for:**
- DLLs loaded from temp/user-writable paths
- Misspelled system DLLs (`amsi.dl`, `ntdl1.dll`)
- Unsigned or unknown DLLs in `System32` processes
- Compare against known-good baseline DLL lists for `lsass.exe`, `services.exe`

## рџ’» Phase 3 вЂ” Command-Line & Console Artifacts

### Step 3.1 вЂ” `cmdscan` (CMD Scanner via CSRSS)

```bash
vol.py -f $DUMP --profile=$PROFILE cmdscan
```

**What it does:** Scans memory for `COMMAND_HISTORY` structures managed by `conhost.exe` / `csrss.exe`. Recovers commands typed into **interactive cmd.exe sessions** вЂ” even after the window was closed.

**What to look for:**
- Manual attacker commands (`net user`, `whoami`, `ipconfig`, `tasklist`)
- File exfiltration commands (`xcopy`, `robocopy`, `ftp`)
- Download commands (`certutil -urlcache`, `bitsadmin`, `curl`)
- Lateral movement (`psexec`, `wmic /node:`)

---

### Step 3.2 вЂ” `consoles` (Full Console I/O with Output)

```bash
vol.py -f $DUMP --profile=$PROFILE consoles
```

**What it does:** Goes further than `cmdscan` вЂ” recovers the full **console screen buffer**, giving you both the commands AND their output. This is like a terminal session replay.

> [!tip] Gold Mine for Post-Exploitation Evidence
> `consoles` can recover output of commands like `net user`, `ipconfig /all`, `dir`, `type <file>` вЂ” the attacker's recon results are preserved here.

**What to look for:**
- Results of `net user /domain`, `whoami /all` (privilege recon)
- Output of `ipconfig`, `arp -a` (network recon)
- LaZagne, Mimikatz output printed to console
- Directory listings of sensitive paths

## рџЊђ Phase 4 вЂ” Network Connections

> [!info] Timeline Note
> `connections` = only **active** connections at dump time  
> `connscan` = active + recently closed (pool scan)

### Step 4.1 вЂ” `connections` (Active TCP Connections вЂ” XP/2003 only)

```bash
vol.py -f $DUMP --profile=$PROFILE connections
```

**What it does:** Reads the `TCPv4` and `TCPv6` connection objects from kernel structures. Returns Local IP:Port, Remote IP:Port, PID, State.

> [!warning] Vista+ Systems
> `connections` does not work on Vista/Win7/Win8/2008. Use `netscan` instead.

---

### Step 4.2 вЂ” `connscan` (Pool Scan for Connections вЂ” XP/2003)

```bash
vol.py -f $DUMP --profile=$PROFILE connscan
```

**What it does:** Pool-based scan for TCP connection objects. Finds connections that were recently closed and whose memory hasn't been zeroed yet.

---

### Step 4.3 вЂ” `netscan` (Comprehensive Network вЂ” Vista/Win7+)

```bash
vol.py -f $DUMP --profile=$PROFILE netscan
```

**What it does:** Scans for `_TCP_ENDPOINT`, `_TCP_LISTENER`, `_UDP_ENDPOINT` pool tags. Returns all active and recently-terminated connections with timestamps, PIDs, protocol.

**What to look for:**
- Connections to unusual external IPs (C2 infrastructure)
- Processes that shouldn't have network access (`lsass.exe`, `spoolsv.exe` making outbound connections)
- High-numbered local ports connecting to low-numbered remote ports (reverse shells: local:49xxx в†’ remote:443/80/4444)
- UDP listeners (DNS tunneling, exfil)
- Connections with no associated PID = terminated process that was communicating

> [!tip] Pivot on IPs found
> Take every remote IP found and cross-reference with [[#Phase 2 вЂ” Process Analysis]] PIDs to understand which process owned the C2 connection.

---

### Step 4.4 вЂ” `sockets` (Open Sockets вЂ” XP only)

```bash
vol.py -f $DUMP --profile=$PROFILE sockets
```

---

### Step 4.5 вЂ” `sockscan` (Pool Scan for Sockets вЂ” XP only)

```bash
vol.py -f $DUMP --profile=$PROFILE sockscan
```

## рџ“¦ Phase 5 вЂ” DLL & Loaded Modules

### Step 5.1 вЂ” `modules` (Kernel Modules / Drivers)

```bash
vol.py -f $DUMP --profile=$PROFILE modules
```

**What it does:** Lists all loaded kernel modules (drivers) via the `PsLoadedModuleList`. Shows module name, base address, size, and full path.

**What to look for:**
- Unknown or unsigned drivers in `\SystemRoot\system32\drivers\`
- Drivers loading from temp paths or user-space paths
- Rootkit drivers (random names, very small sizes, no path)

---

### Step 5.2 вЂ” `modscan` (Pool Scan for Kernel Modules)

```bash
vol.py -f $DUMP --profile=$PROFILE modscan
```

**What it does:** Pool scan equivalent of `modules`. Finds **unlinked kernel modules** вЂ” i.e., drivers a rootkit has hidden by removing from `PsLoadedModuleList`.

> [!important] Cross-reference `modules` vs `modscan`
> Modules in `modscan` but not in `modules` = **kernel-level rootkit hiding drivers**

---

### Step 5.3 вЂ” `ldrmodules` (PEB Loader Cross-Check)

```bash
vol.py -f $DUMP --profile=$PROFILE ldrmodules -p 1234
```

**What it does:** Checks three linked lists in the PEB loader structure (`InLoad`, `InInit`, `InMem`) for each process. Legitimate DLLs appear in all three; injected/manually-mapped code often appears in only one or zero.

**What to look for:**
- `False False False` for `InLoad`, `InInit`, `InMem` = **manually mapped / injected DLL** (no loader record at all)
- `False False True` or similar inconsistencies
- This is how **reflective DLL injection** and **process hollowing** leave traces

## рџ¦  Phase 6 вЂ” Code Injection & Hollowing Detection

> [!danger] This phase = malware hunting
> Most malware will leave evidence here. Always run these after establishing baseline process/network picture.

### Step 6.1 вЂ” `malfind` (Find Injected Code)

```bash
vol.py -f $DUMP --profile=$PROFILE malfind
```

**What it does:** Finds **suspicious memory regions** within processes by looking for:
- `PAGE_EXECUTE_READWRITE` (RWX) permissions вЂ” writable + executable = injection staging area
- VAD (Virtual Address Descriptor) entries that have no backing file on disk
- Regions containing PE headers (`MZ` / `4D5A` at offset 0)

**What to look for:**
- Any process with injected PE (shows disassembly of the first bytes)
- Shellcode patterns (look for `CALL/JMP` with immediate offsets in the disassembly snippet)
- High base addresses injected into `explorer.exe`, `svchost.exe`, `notepad.exe`

```bash
# Dump suspicious regions to files for further analysis
vol.py -f $DUMP --profile=$PROFILE malfind -D /tmp/malfind_output/
```

---

### Step 6.2 вЂ” `hollowfind` (Process Hollowing Detector вЂ” community plugin)

```bash
vol.py --plugins=/opt/community/ -f $DUMP --profile=$PROFILE hollowfind
```

**What it does:** Specifically detects **process hollowing** вЂ” where legitimate process memory is unmapped and replaced with malicious code. Checks for mismatches between:
- PEB `ImageBaseAddress` and the actual mapped PE at that address
- Module base in VAD vs loaded module list

---

### Step 6.3 вЂ” `apihooks` (Detect API Hook Trampolines)

```bash
# All processes (slow)
vol.py -f $DUMP --profile=$PROFILE apihooks

# Specific process
vol.py -f $DUMP --profile=$PROFILE apihooks -p 1234
```

**What it does:** Enumerates every exported function in loaded DLLs and checks if the first bytes have been **modified with a JMP/CALL trampoline** pointing elsewhere (userland hooking technique used by rootkits, AV, and malware alike).

**What to look for:**
- Hooks in `ntdll.dll` exports в†’ inline hooking of native API (syscall interception)
- Hooks in `ws2_32.dll` в†’ network function hooking (traffic interception/redirection)
- Hooks pointing into anonymous/non-module memory regions = malicious hooks

---

### Step 6.4 вЂ” `vadinfo` (Virtual Address Descriptor Tree)

```bash
vol.py -f $DUMP --profile=$PROFILE vadinfo -p 1234
```

**What it does:** Dumps the full VAD tree for a process вЂ” every virtual memory region with its permissions, type, and backing file.

**What to look for:**
- `Private` + `PAGE_EXECUTE_READWRITE` with no mapped file в†’ shellcode/injection
- Very large private executable regions in otherwise simple processes

---

### Step 6.5 вЂ” `vaddump` (Dump VAD Regions)

```bash
vol.py -f $DUMP --profile=$PROFILE vaddump -p 1234 -D /tmp/vaddumps/
```

**What it does:** Extracts all virtual memory regions from a specific process to disk for offline analysis with strings, IDA, Ghidra.

## рџ—‚пёЏ Phase 7 вЂ” Handles & Open Files

### Step 7.1 вЂ” `handles` (Open Handles per Process)

```bash
# All types
vol.py -f $DUMP --profile=$PROFILE handles -p 1234

# Only File handles
vol.py -f $DUMP --profile=$PROFILE handles -p 1234 -t File

# Only Registry key handles
vol.py -f $DUMP --profile=$PROFILE handles -p 1234 -t Key

# Only Mutant (mutex) handles
vol.py -f $DUMP --profile=$PROFILE handles -p 1234 -t Mutant
```

**What it does:** Lists all kernel objects a process has open handles to. Very powerful for:
- Finding what files were being read/written at dump time
- Finding C2 or persistence-related registry keys
- Finding **mutex names** вЂ” malware families use hardcoded mutexes to prevent double-infection

> [!tip] Mutex = Malware Fingerprint
> Common malware mutexes:
> - `Global\{GUID}` style = often malware
> - `PoisonIvy`, `njRAT`, `DarkComet` have known mutex patterns
> - Search extracted mutex names on VirusTotal / MalwareBazaar

---

### Step 7.2 вЂ” `filescan` (Pool Scan for FILE_OBJECT Structures)

```bash
vol.py -f $DUMP --profile=$PROFILE filescan
```

**What it does:** Scans memory for `_FILE_OBJECT` pool tag to enumerate every file that was open (or recently open) in the system вЂ” regardless of whether the handle is still held.

**What to look for:**
- Interesting file paths: `\Device\HarddiskVolume\Temp\`, `\Users\<user>\AppData\Roaming\`
- Suspicious file names in system directories
- `.exe`, `.dll` in temp or user paths
- Evidence files like `.zip`, `.rar` containing staged exfil data

```bash
# Filter for executables
vol.py -f $DUMP --profile=$PROFILE filescan | grep -iE "\.exe|\.dll|\.bat|\.ps1|\.vbs"

# Filter for a specific user
vol.py -f $DUMP --profile=$PROFILE filescan | grep -i "Users\\\\john"
```

---

### Step 7.3 вЂ” `dumpfiles` (Extract Files from Memory)

```bash
# Dump by physical address (from filescan output)
vol.py -f $DUMP --profile=$PROFILE dumpfiles -Q 0x000000007e7a3a30 -D /tmp/dumpfiles/

# Dump all files matching a regex
vol.py -f $DUMP --profile=$PROFILE dumpfiles --regex="\.exe$" -i -D /tmp/dumpfiles/
```

**What it does:** Extracts file data from the memory-resident **cache manager pages** and **VAD-mapped sections**. This lets you recover files that were in use at the time of the dump even if they were deleted from disk.

> [!important] Two Output Types
> - `file.*.vacb` = from Cache Manager (system cache)
> - `file.*.dat` = from VAD-mapped section
> Run `file` command on extracted outputs to confirm PE/archive/document type.

## рџ—ќпёЏ Phase 8 вЂ” Registry Forensics

### Step 8.1 вЂ” `hivelist` (List Loaded Registry Hives)

```bash
vol.py -f $DUMP --profile=$PROFILE hivelist
```

**What it does:** Lists all registry hive files currently loaded in memory, with their virtual address and file path. You need these addresses for all subsequent registry commands.

**Key hives to note:**
| Hive | Path | Contains |
|------|------|----------|
| SYSTEM | `\REGISTRY\MACHINE\SYSTEM` | Boot config, services, drivers |
| SOFTWARE | `\REGISTRY\MACHINE\SOFTWARE` | Installed software, OS settings |
| SAM | `\REGISTRY\MACHINE\SAM` | Local user hashes |
| SECURITY | `\REGISTRY\MACHINE\SECURITY` | LSA secrets, cached creds |
| NTUSER.DAT | `\...\<user>\ntuser.dat` | Per-user settings, run keys |

---

### Step 8.2 вЂ” `printkey` (Read Registry Keys)

```bash
# Root of a hive (use virtual address from hivelist)
vol.py -f $DUMP --profile=$PROFILE printkey -o 0xfffff8a000023010

# Specific key path
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Windows\CurrentVersion\Run"

# Per-user startup (autorun)
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce"
```

**What it does:** Reads and displays registry key values at a specified path, including timestamps.

**Critical keys to check:**
```bash
# Autoruns
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Windows\CurrentVersion\Run"
vol.py -f $DUMP --profile=$PROFILE printkey -K "SYSTEM\CurrentControlSet\Services"

# Recent files / execution evidence
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs"
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\RunMRU"

# USB devices
vol.py -f $DUMP --profile=$PROFILE printkey -K "SYSTEM\CurrentControlSet\Enum\USBSTOR"

# Typed URLs
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Internet Explorer\TypedURLs"

# UserAssist (GUI program execution вЂ” ROT13 encoded)
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\UserAssist"

# MRU (Most Recently Used command in Run dialog)
vol.py -f $DUMP --profile=$PROFILE printkey -K "SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\RunMRU"

# AppCompatCache (ShimCache вЂ” execution evidence)
vol.py -f $DUMP --profile=$PROFILE printkey -K "SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache"
```

---

### Step 8.3 вЂ” `userassist` (GUI Execution Artifacts)

```bash
vol.py -f $DUMP --profile=$PROFILE userassist
```

**What it does:** Parses the `UserAssist` registry keys (which track GUI program executions) and decodes the ROT13-encoded entries automatically. Shows program paths, run counts, and last execution time.

---

### Step 8.4 вЂ” `shellbags` (Folder Navigation History)

```bash
vol.py -f $DUMP --profile=$PROFILE shellbags
```

**What it does:** Parses `BagMRU`/`Bags` registry keys which record folders the user has navigated to in Windows Explorer. Useful for proving access to specific directories even if file system records were wiped.

---

### Step 8.5 вЂ” `hivedump` (Dump Entire Hive to File)

```bash
vol.py -f $DUMP --profile=$PROFILE hivedump -o 0xfffff8a000023010 -w /tmp/SYSTEM.hiv
```

**What it does:** Exports a complete registry hive from memory to a `.hiv` file that can be loaded into `regedit`, RegRipper, or other tools for deep offline analysis.

## рџ“‚ Phase 9 вЂ” File System Artifacts

### Step 9.1 вЂ” `mftparser` (Master File Table Parser)

```bash
vol.py -f $DUMP --profile=$PROFILE mftparser
```

**What it does:** Parses the **$MFT** (Master File Table) resident in memory. Recovers file metadata including:
- Standard Information timestamps (Created, Modified, Changed, Accessed)
- Filename timestamps (harder to stomp)
- Resident file data (small files are stored directly in the MFT entry)
- Deleted file entries

```bash
# Filter for recently modified files
vol.py -f $DUMP --profile=$PROFILE mftparser | grep -A5 "2024-03-15"

# Find deleted files
vol.py -f $DUMP --profile=$PROFILE mftparser | grep -i "DELETED"
```

---

### Step 9.2 вЂ” `shimcache` (AppCompatCache / Execution Evidence)

```bash
vol.py -f $DUMP --profile=$PROFILE shimcache
```

**What it does:** Parses `AppCompatCache` (ShimCache) from the SYSTEM hive in memory. Records every executable that was run or present on the system, with path, size, and last modified time. Provides **execution evidence** even for deleted malware.

---

### Step 9.3 вЂ” `prefetchparser` (Prefetch Files)

```bash
vol.py -f $DUMP --profile=$PROFILE prefetchparser
```

**What it does:** Parses Windows prefetch files that may be loaded in memory. Prefetch records execution of programs with timestamps and loaded file lists.

## рџ”‘ Phase 11 вЂ” Credential Harvesting

> [!danger] Legal Reminder
> Only run credential extraction on systems you have **written authorization** for. These commands recover actual password hashes and plaintext credentials.

### Step 11.1 вЂ” `hashdump` (SAM Database вЂ” Local User NTLM Hashes)

```bash
vol.py -f $DUMP --profile=$PROFILE hashdump
```

**What it does:** Extracts NTLM password hashes for all local user accounts from the SAM database. Requires both the SAM and SYSTEM hives to be loaded (the SYSKEY boot key is in SYSTEM).

**Output format:**
```
Administrator:500:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
john.doe:1001:aad3b435b51404eeaad3b435b51404ee:e3d5a71bd4afc4f3e7e0e72ebe6cadb5:::
```

> [!tip] Crack with Hashcat
> ```bash
> hashcat -m 1000 hashes.txt /usr/share/wordlists/rockyou.txt
> ```

---

### Step 11.2 вЂ” `cachedump` (Cached Domain Credentials вЂ” MSCache)

```bash
vol.py -f $DUMP --profile=$PROFILE cachedump
```

**What it does:** Extracts Domain Cached Credentials (DCC2/MSCachev2) from the `SECURITY\Cache` registry key. These are cached when a domain user logs into a machine that can't reach the DC.

```bash
# Crack with Hashcat (DCC2 = mode 2100)
hashcat -m 2100 cached_hashes.txt rockyou.txt
```

---

### Step 11.3 вЂ” `lsadump` (LSA Secrets)

```bash
vol.py -f $DUMP --profile=$PROFILE lsadump
```

**What it does:** Decrypts LSA (Local Security Authority) Secrets from the `SECURITY` registry hive. Can contain:
- Service account plaintext passwords
- Machine account passwords
- Cached credentials for scheduled tasks and auto-login accounts
- `_SC_<ServiceName>` entries = service account passwords in plaintext

---

### Step 11.4 вЂ” `mimikatz` (Volatility Mimikatz Plugin)

```bash
vol.py --plugins=/opt/community/ -f $DUMP --profile=$PROFILE mimikatz
```

**What it does:** Ports Mimikatz's `sekurlsa::logonpasswords` logic into Volatility. Extracts credentials from `lsass.exe` memory:
- NTLM hashes
- SHA1 hashes
- **Plaintext passwords** (on older systems / pre-KB2871997 patch)
- Kerberos tickets (TGTs, service tickets)
- WDigest plaintext credentials

> [!note] For newer Win8.1+/2012R2+ systems
> Plaintext credentials are rarely available due to WDigest disable. But NTLM hashes are still recovered.

## рџ”„ Phase 12 вЂ” Persistence Mechanisms

### Step 12.1 вЂ” `svcscan` (Windows Services)

```bash
vol.py -f $DUMP --profile=$PROFILE svcscan
```

**What it does:** Scans for `_SERVICE_RECORD` structures to enumerate all Windows services вЂ” including **stopped** ones and those hidden from the SCM. Returns service name, display name, binary path, state, PID.

**What to look for:**
- Services running from non-standard paths (`%TEMP%`, `C:\Users\`)
- Services with suspicious names (random strings, lookalike names)
- Services with `Start=2` (auto-start) pointing to unknown binaries
- Services with no description or display name (malware often skips these)

- Registry Run/RunOnce keys (HKLM + HKCU)

## рџ›ЎпёЏ Phase 14 вЂ” Kernel & Rootkit Analysis

### Step 14.1 вЂ” `ssdt` (System Service Descriptor Table Hooks)

```bash
vol.py -f $DUMP --profile=$PROFILE ssdt
```

**What it does:** Dumps the SSDT вЂ” the kernel dispatch table that maps syscall numbers to handler functions. Checks whether each entry points to `ntoskrnl.exe`/`win32k.sys` (legitimate) or somewhere else (hooked by rootkit).

**What to look for:**
- SSDT entries pointing outside `ntoskrnl.exe` address range = kernel-mode rootkit hook
- Common target syscalls: `NtOpenProcess`, `NtQuerySystemInformation`, `NtCreateFile`

---

### Step 14.2 вЂ” `idt` (Interrupt Descriptor Table)

```bash
vol.py -f $DUMP --profile=$PROFILE idt
```

**What it does:** Dumps the IDT for each CPU. Checks if interrupt handlers have been redirected to malicious code (kernel rootkit technique).

---

### Step 14.3 вЂ” `gdt` (Global Descriptor Table)

```bash
vol.py -f $DUMP --profile=$PROFILE gdt
```

**What it does:** Dumps the GDT. Useful for detecting call gate hooking (advanced rootkit technique on 32-bit systems).

---

### Step 14.4 вЂ” `driverirp` (Driver IRP Hooks)

```bash
vol.py -f $DUMP --profile=$PROFILE driverirp
```

**What it does:** Checks the IRP (I/O Request Packet) dispatch tables of all loaded drivers for hooks. Rootkits commonly hook IRP handlers in `\FileSystem\` or `\Driver\Disk` drivers to intercept file I/O.

---

### Step 14.5 вЂ” `timers` (Kernel Timer Objects)

```bash
vol.py -f $DUMP --profile=$PROFILE timers
```

**What it does:** Enumerates all `_KTIMER` objects in the kernel pool. Rootkits sometimes use timers to re-hook SSDT entries after they've been unhooked.

---

### Step 14.6 вЂ” `callbacks` (Kernel Notify Callbacks)

```bash
vol.py -f $DUMP --profile=$PROFILE callbacks
```

**What it does:** Enumerates all registered kernel callback routines вЂ” `PsSetCreateProcessNotifyRoutine`, `PsSetLoadImageNotifyRoutine`, etc. EDR/AV register callbacks here, but so do rootkits.

**What to look for:**
- Callbacks registered to unknown/unsigned drivers
- `ObRegisterCallbacks` entries that can block process handle access

## рџљ© Cheat Sheet вЂ” Common Suspicious Indicators

> [!warning] Red Flags in Memory Forensics
> - `cmd.exe` / `powershell.exe` spawned by Office, browser, or PDF process
> - `lsass.exe` with memory regions showing `MZ` header (mimikatz injection)
> - `svchost.exe` NOT parented by `services.exe`
> - `explorer.exe` with outbound network connection to external IP
> - Process with no path / blank image file name in pslist
> - SSDT entries pointing outside ntoskrnl address range
> - `PAGE_EXECUTE_READWRITE` private memory regions with MZ headers (malfind hits)
> - Services running from `C:\Users\`, `%TEMP%`, `%APPDATA%`
> - NTLM hashes for accounts that shouldn't be local admins
> - Clipboard containing credentials or base64 blobs
> - Chrome history showing exfil sites or C2 domains
> - Console output showing `sekurlsa::logonpasswords` or LaZagne output
