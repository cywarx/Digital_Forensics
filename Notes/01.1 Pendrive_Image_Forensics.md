> [!tip] Goal
> Create a 32 MB virtual pendrive image **with a proper partition table** for forensics practice — compatible with `Autopsy`, `Sleuth Kit`, `fls`, `icat`, and `foremost`.

---

## Why Add a Partition Table?

> [!warning]- Bare FAT image vs Proper Disk Image
> A bare `mkfs.fat` image has **no MBR/partition table** — just a raw filesystem.
> Autopsy expects a real disk structure:
>
> | Image Type | `fls` works? | Autopsy works? |
> |---|---|---|
> | Bare FAT (no partition table) | ✅ Yes | ⚠️ Only as "Logical" |
> | MBR + FAT partition | ✅ Yes | ✅ Yes (as "Disk") |
>
> Adding a partition table makes your image behave exactly like a real USB drive.

---

## Step 1 — Create the Image

```bash
dd if=/dev/zero of=pendrive.img bs=1M count=32
```

> [!info]- How does this make 32 MB?
> - `bs=1M` → each block = **1 MB**
> - `count=32` → write **32 blocks**
> - `1 MB × 32 = 32 MB` total image

---

## Step 2 — Hash with dc3dd (Forensic Integrity)

```bash
dc3dd if=pendrive.img hash=md5 hash=sha256 log=pendrive.log
```

> [!info]- Why hash before doing anything?
> Generates **MD5 and SHA-256** cryptographic hashes to prove the image has not been tampered with — essential for forensic chain of custody.
>
> | Option | Meaning |
> |---|---|
> | `hash=md5` | Generate MD5 hash |
> | `hash=sha256` | Generate SHA-256 hash |
> | `log=pendrive.log` | Save both hashes to a single log file |
>
> > Always hash **before** and **after** analysis to prove integrity.

> [!info]- MD5 vs SHA-256 — which to use?
>
> | Hash | Speed | Length | Used For |
> |---|---|---|---|
> | MD5 | ⚡ Fast | 128-bit (32 chars) | Quick verification, legacy tools |
> | SHA-256 | Slower | 256-bit (64 chars) | Court-admissible, modern forensics |
>
> Using **both** is best practice — MD5 for speed, SHA-256 for integrity.

> [!tip]- Verify hash after analysis
> After you finish analysis, re-hash and compare:
> ```bash
> dc3dd if=pendrive.img hash=md5 hash=sha256 log=pendrive_after.log
> diff pendrive.log pendrive_after.log
> ```
> If hashes match → image was not modified ✅

---

## Step 3 — Create Partition Table

```bash
parted pendrive.img --script mklabel msdos mkpart primary fat16 1MiB 31MiB
```

> [!info]- What does this do?
> Creates a real **MBR partition table** with one FAT16 partition inside the image — exactly like a real USB drive.
>
> ```
> pendrive.img layout after this step:
> ┌─────────────────────────────────────┐
> │  MBR (first 1 MiB)                 │
> ├─────────────────────────────────────┤
> │  FAT16 Partition (1 MiB → 31 MiB)  │
> └─────────────────────────────────────┘
> ```
>
> | Option | Meaning |
> |---|---|
> | `mklabel msdos` | Create MBR partition table |
> | `mkpart primary fat16` | Add one primary FAT16 partition |
> | `1MiB 31MiB` | Partition starts at 1 MiB, ends at 31 MiB |

---

## Step 4 — Format the Partition as FAT16

```bash
mkfs.fat -F 16 -n "HEXX_USB" --offset 2048 pendrive.img
```

> [!info]- Why `--offset 2048`?
> The partition starts at **1 MiB = 2048 sectors × 512 bytes**.
> Without the offset, `mkfs.fat` would overwrite the MBR partition table.
>
> | Value | Meaning |
> |---|---|
> | `--offset 2048` | Start formatting at sector 2048 (= 1 MiB) |
> | `-F 16` | Force FAT16 format |
> | `-n "HEXX_USB"` | Set volume label |

> [!info]- Why does pendrive_mount show Jan 1 1970 after formatting?
> FAT filesystems do not store a creation date for the root directory — defaults to **Unix Epoch (Jan 1 1970)**. Completely normal.
>
> | Filesystem | Tracks folder dates? |
> |--|--|
> | FAT16 / FAT32 | ❌ No — defaults to Jan 1 1970 |
> | ext4 | ✅ Yes — shows real timestamp |

## Step 5 — Mount

```bash
mkdir pendrive_mount
sudo mount -o loop,offset=1048576,uid=1000,gid=1000 pendrive.img pendrive_mount/
```

> [!info]- Why `offset=1048576`?
> The partition starts at **1 MiB = 1,048,576 bytes**.
> Without the offset, mount would see the MBR instead of the FAT partition.

> [!warning]- RO Flag — Read-Only vs Read-Write
> Check after mounting:
> ```bash
> lsblk
> ```
> - `RO=1` → Read-Only — force read-write:
> ```bash
> sudo mount -o loop,offset=1048576,rw,uid=1000,gid=1000 pendrive.img pendrive_mount/
> ```

> [!warning]- Why does sudo change ownership of pendrive_mount?
> Mounting with `sudo` makes **root** the owner. Fix with `uid/gid` flags:
> ```bash
> sudo mount -o loop,offset=1048576,uid=1000,gid=1000 pendrive.img pendrive_mount/
> ```
> Check your uid/gid with `id`.

> [!info]- What is `-o loop`?
> A **loop device** lets you mount a regular file as if it were a physical disk.
>
> **Normal disk:**
> ```
> Physical USB → /dev/sdb → mount → access files
> ```
>
> **Loop device:**
> ```
> pendrive.img → /dev/loop0 → mount → access files
> ```
>
> `-o loop` = *"treat this file as a disk"*

> [!info]- Leftover loop devices — how to remove
> Sometimes old loop devices stay attached even after unmounting. Check with:
> ```bash
> sudo losetup -l
> ```
> Detach them:
> ```bash
> sudo losetup -d /dev/loop0
> ```
> If still showing in file explorer, refresh Nautilus:
> ```bash
> nautilus -q
> ```

## Step 7 — Delete Files (For Recovery Practice)

```bash
rm pendrive_mount/secret_notes.txt
rm pendrive_mount/flag.ctf
```

> [!info]- What happens when you delete on FAT?
> On FAT16/FAT32, deleting a file only marks the directory entry with `0xE5` — the actual data **remains on disk** until overwritten.
>
> This is why forensic recovery works — `fls` will show deleted files with a `*` prefix:
> ```
> r/r   4:  readme.txt         ← visible file
> r/r * 15: secret_notes.txt   ← deleted but recoverable
> r/r * 22: flag.ctf           ← deleted but recoverable
> ```

---

## Step 8 — Unmount

```bash
sudo umount pendrive_mount/
```

> [!warning]- Always unmount before forensic analysis
> Never run Autopsy or Sleuth Kit on a mounted image — always unmount first to avoid filesystem inconsistencies.

---

## Step 9 — Verify with Sleuth Kit

```bash
# Check partition table exists
mmls pendrive.img

# Check filesystem details
fsstat -o 2048 pendrive.img

# List all files including deleted (marked with *)
fls -o 2048 pendrive.img

# List only deleted files
fls -d -o 2048 pendrive.img

# Recover a deleted file by inode number
icat -o 2048 pendrive.img <inode_number> > recovered_file
```

> [!info]- Why `-o 2048` in Sleuth Kit commands?
> Because the FAT16 partition starts at sector 2048 (1 MiB offset).
> Without `-o 2048`, Sleuth Kit reads the MBR and gets confused.
