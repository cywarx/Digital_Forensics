What is Email Forensics?

Email forensics is the systematic process of collecting, analyzing, and preserving email evidence to investigate cybercrimes, fraud, phishing, harassment, or unauthorized access. It involves tracing email origins, verifying authenticity, and building timelines.

Real Header Example

Below is a real-world raw email header from a suspicious email. Click each section to understand what it means:

Received: Hop Chain (Most Important — Read Bottom to Top) Critical
Received: from mail.evil-sender.ru (185.220.101.45) by mx.victim-corp.com with ESMTP id x7si123 for <victim@victim-corp.com>; Thu, 15 May 2025 09:14:22 -0700 # ← This is the LAST hop (nearest to recipient = least trustworthy) # "from" = sending server self-identified name # IP in () = ACTUAL IP address (trust this over the name) # "by" = receiving mail server Received: from webmail.evil-sender.ru (185.220.101.45) by mail.evil-sender.ru with ESMTPS Thu, 15 May 2025 09:13:55 +0300 # ← This is the FIRST hop (origin of the email) # Read from BOTTOM to TOP for true path
Key Rule

The BOTTOM-MOST "Received" header is where the email originated. Each hop ABOVE is a relay. Attackers CAN forge headers above — trust only the first hop received by your own mail server.

From / Return-Path / Reply-To: Identity Fields Spoofable
From: "PayPal Security" <noreply@paypa1.com> Reply-To: harvest@phisher.ru Return-Path: <bounce@evil-mailer.net> # ANALYSIS: # From: display name says "PayPal" but domain is paypa1.com (number 1 not letter l!) # Reply-To: replies go to attacker, not sender # Return-Path: bounces (delivery failures) go to attacker's server # These fields are ENTIRELY under the sender's control — always verify!
Message-ID / X-Mailer: Origin Fingerprinting
Message-ID: <CABc123xyz@mail.gmail.com> # Legitimate: domain matches sender's domain # Suspicious: Message-ID domain differs from From: domain Message-ID: <20250515091422.12345@evil-sender.ru> # Reveals real sending server: evil-sender.ru X-Mailer: The Bat! 10.3.1.1 # Reveals the email client used (can be faked) X-Originating-IP: 185.220.101.45 # Added by some webmail providers — the device's public IP # Gmail stopped including this; Yahoo/Hotmail still do sometimes
Date / Timezone: Timestamp Analysis
Date: Thu, 15 May 2025 09:14:22 -0700 # Timezone offset: -0700 = PDT (Pacific Daylight Time, USA) # BUT: The Received header shows +0300 (Moscow/Istanbul time) # MISMATCH: Date claims US Pacific, server is Eastern Europe # This timezone inconsistency is a RED FLAG # Compare ALL timestamps in Received chain: Received: ... Thu, 15 May 2025 09:13:55 +0300 Received: ... Thu, 15 May 2025 09:14:01 +0300 Received: ... Thu, 15 May 2025 09:14:22 -0700 # Transit time between hops should be 0–120 seconds. Longer = suspicious.

SPF — Sender Policy Framework

SPF is a DNS TXT record that lists which IP addresses are authorized to send email for a domain. The receiving mail server checks if the sending IP is in this list.

How it works

When email arrives from user@example.com, the receiving server looks up the DNS TXT record for example.com and checks if the sending IP is authorized there.

SPF DNS Record Example

# DNS lookup: dig TXT example.com example.com. IN TXT "v=spf1 ip4:192.0.2.0/24 ip4:198.51.100.123 include:_spf.google.com ~all" # v=spf1 → SPF version 1 # ip4:192.0.2.0/24 → Entire /24 range is authorized # ip4:198.51.100.123 → This specific IP is authorized # include:_spf.google.com → Also authorize Google's mail servers # ~all → SoftFail: anything else is suspicious (not a hard reject) # -all → HardFail: anything else MUST be rejected # +all → Allow ALL (dangerous — do not use)

SPF Result in Email Header

Received-SPF: pass (google.com: domain of support@amazon.com designates 192.0.2.47 as permitted sender) client-ip=192.0.2.47; ─── Phishing email example ─── Received-SPF: fail (google.com: domain of noreply@amazon.com does NOT designate 185.220.101.45 as permitted sender) client-ip=185.220.101.45; mechanism="-all"
✓ pass — IP is authorized ✗ fail — IP not authorized, reject ~ softfail — suspicious but accept ? neutral — no policy

DKIM — DomainKeys Identified Mail

DKIM adds a cryptographic digital signature to emails. The signing domain publishes its public key in DNS. Recipients verify the signature to confirm the email wasn't modified in transit.

Why DKIM matters

SPF only checks if the IP is authorized. DKIM proves the email content was not tampered with after the sender signed it. Even if someone relays your email, a valid DKIM signature proves the body is intact.

DKIM Signature in Header

DKIM-Signature: v=1; a=rsa-sha256; c=relaxed/relaxed; d=amazon.com; ← signing domain s=20210112; ← selector (key identifier) h=from:to:subject:date:message-id:mime-version; ↑ which headers are covered by signature bh=Yc1QVqPdZWwDvhWqaEn...; ← hash of body b=ABCDef123...GHIJkl456== ← actual digital signature # DNS key lookup: dig TXT 20210112._domainkey.amazon.com 20210112._domainkey.amazon.com. IN TXT "v=DKIM1; k=rsa; p=MIGfMA0..." ─── Forged email — DKIM fails ─── Authentication-Results: dkim=fail (signature did not verify) header.d=amazon.com header.s=20210112 # Body was modified after signing, or header.d doesn't match From domain
DKIM Replay Attack

An attacker can copy a legitimate DKIM-signed email and replay it to new victims. The signature still passes! This is why DMARC alignment checks are also needed.

DMARC — Domain-based Message Authentication

DMARC ties SPF and DKIM together. It tells receivers what to do when SPF or DKIM fail, and requires that the "From" domain aligns with SPF/DKIM domains.

# DNS lookup: dig TXT _dmarc.example.com _dmarc.example.com. IN TXT "v=DMARC1; p=reject; ← policy: none / quarantine / reject pct=100; ← apply to 100% of messages rua=mailto:dmarc-reports@example.com; ← send aggregate reports here ruf=mailto:forensics@example.com; ← send forensic reports here aspf=s; ← strict SPF alignment adkim=s" ← strict DKIM alignment # Policies: # none = monitor only, take no action (bad — lets phishing through) # quarantine = send to spam folder # reject = refuse delivery entirely (best protection)

DMARC Result in Header

─── Legitimate email ─── Authentication-Results: mx.google.com; spf=pass (google.com: domain of @amazon.com designates 192.0.2.47 as permitted) dkim=pass header.i=@amazon.com header.s=20210112 dmarc=pass (p=REJECT sp=REJECT dis=NONE) header.from=amazon.com ─── Phishing/spoofed email ─── Authentication-Results: mx.google.com; spf=fail (domain of noreply@amazon.com does not designate 185.220.101.45) dkim=fail dmarc=fail (p=REJECT sp=REJECT dis=REJECT) header.from=amazon.com # Result: email should have been REJECTED but might still slip through # if recipient server ignores DMARC policy (common with weak configs)

Here is a complete Authentication-Results header from a real phishing email impersonating a bank:

Authentication-Results: mx.google.com; dkim=fail reason="signature verification failed" header.i=@secure-hsbc-alerts.com header.s=default header.b=aB3xY7pQ; spf=softfail (google.com: domain of alert@secure-hsbc-alerts.com does not designate 45.33.32.156 as permitted sender) smtp.mailfrom=alert@secure-hsbc-alerts.com; dmarc=fail (p=NONE sp=NONE dis=NONE) header.from=secure-hsbc-alerts.com ─── Forensic Analysis ─── # 1. Domain secure-hsbc-alerts.com is NOT hsbc.com. SPF softfail + DKIM fail + DMARC fail = clear phishing attempt. The sender registered a lookalike domain to bypass naive From: header checks.

Domain Spoofing Techniques

Homograph / Lookalike Domain Attack
# Real domain vs. fake domains (visual comparison): paypal.com ← Legitimate paypa1.com ← Number 1 instead of letter l paypal-secure.com ← Extra subdomain prepended pаypal.com ← Cyrillic 'а' (Unicode U+0430) looks identical to Latin 'a' paypa⁠l.com ← Zero-width space inserted (invisible!) # Check for IDN homograph: $ python3 -c "print('pаypal.com'.encode('idna'))" # Output: xn--pypal-4ve.com ← Punycode reveals Cyrillic character!
Display Name Spoofing
# Email clients show display name prominently From: "Apple Support <noreply@apple.com>" <phisher@random-server.ru> # The display name is "Apple Support <noreply@apple.com>" # Many clients show this as the sender — including the fake Apple address # The REAL sending address is phisher@random-server.ru # Another common form: From: "security-alert@amazon.com" <no-reply@amaz0n-support.net>
Malicious Link Analysis
# Phishing email body HTML source: <a href="https://secure-login.paypal.com.verification-needed.xyz/auth"> Click here to verify your account </a> # URL Analysis: # Domain is: verification-needed.xyz (NOT paypal.com!) # paypal.com appears in the PATH, not the domain # Browser reads: [subdomain].[domain].[TLD]/[path] # = secure-login.paypal.com . verification-needed.xyz / auth # Defang URLs before analysis (prevent accidental clicks): $ python3 -c " url = 'https://secure-login.paypal.com.verification-needed.xyz/auth' print(url.replace('http', 'hxxp').replace('.', '[.]')) " # Output: hxxps://secure-login[.]paypal[.]com[.]verification-needed[.]xyz/auth

Social Engineering Indicators

Urgency / Scarcity

Phrases like "Your account will be closed in 24 hours", "Immediate action required", "Last warning". Creates panic to bypass critical thinking.

Rewards / Too Good

"You've won $1,000", "Unclaimed package", "Tax refund of $450 awaiting". Exploits greed and curiosity.

Authority

Impersonating CEO (BEC), bank, IRS/IT department, police. "This is the IT department, your password must be reset now."

Obfuscation

Tiny font URLs, white text on white background, HTML comments breaking keywords to bypass filters, base64-encoded content.

Attachment Analysis Malware Detection
Safely examine suspicious email attachments without executing them. Extract metadata, hashes, and malware indicators.
Safety First

NEVER open suspicious attachments on your main machine. Use an isolated VM (Kali Linux, REMnux) or an online sandbox like any.run, Joe Sandbox, or Triage. Disable network access before analyzing.

MIME Decoding — Extract Attachments from Raw Email

# Raw email MIME structure: MIME-Version: 1.0 Content-Type: multipart/mixed; boundary="==_boundary_12345==" --==_boundary_12345== Content-Type: text/html; charset="UTF-8" [HTML email body here] --==_boundary_12345== Content-Type: application/octet-stream; name="Invoice_May2025.pdf.exe" Content-Disposition: attachment; filename="Invoice_May2025.pdf.exe" Content-Transfer-Encoding: base64 TVqQAAMAAAAEAAAA//8AALgAAAAAAAAAQA... # ↑ Base64 encoded executable! # Filename shows .pdf.exe — Windows hides .exe extension by default # Victim sees "Invoice_May2025.pdf" with PDF icon

Python — Extract and Analyze Attachments

import email, hashlib, magic, sys # Load raw email with open('suspicious.eml', 'rb') as f: msg = email.message_from_binary_file(f) for part in msg.walk(): if part.get_content_disposition() == 'attachment': filename = part.get_filename() payload = part.get_payload(decode=True) # Calculate hashes md5 = hashlib.md5(payload).hexdigest() sha1 = hashlib.sha1(payload).hexdigest() sha256 = hashlib.sha256(payload).hexdigest() # Detect true file type (ignore extension) file_type = magic.from_buffer(payload, mime=True) print(f"Filename: {filename}") print(f"True type: {file_type}") ← CRITICAL: compare to extension print(f"MD5: {md5}") print(f"SHA256: {sha256}") ─── Output ─── Filename: Invoice_May2025.pdf True type: application/x-dosexec ← It's an EXE disguised as PDF! MD5: a1b2c3d4e5f6789012345678 SHA256: deadbeef1234...abc789 ← Submit this to VirusTotal

Python Analysis Script Template

#!/usr/bin/env python3 """Email Forensic Analysis Script — Use on isolated system""" import email, hashlib, re, socket from email import policy def analyze_email(filepath): with open(filepath, 'rb') as f: msg = email.message_from_binary_file(f, policy=policy.default) print("=== HEADER ANALYSIS ===") print(f"From: {msg['from']}") print(f"Reply-To: {msg['reply-to']}") print(f"Subject: {msg['subject']}") print(f"Date: {msg['date']}") print(f"Msg-ID: {msg['message-id']}") print("\n=== RECEIVED HOPS ===") for received in msg.get_all('received', []): ips = re.findall(r'\b(?:\d{1,3}\.){3}\d{1,3}\b', received) print(f" IPs: {ips}") print(f" {received[:80]}...") print("\n=== AUTHENTICATION ===") auth = msg.get('authentication-results', 'NOT FOUND') print(auth) print("\n=== ATTACHMENTS ===") for part in msg.walk(): if part.get_content_disposition() == 'attachment': name = part.get_filename() data = part.get_payload(decode=True) sha256 = hashlib.sha256(data).hexdigest() print(f" File: {name}") print(f" SHA256: {sha256}") print(f" Size: {len(data)} bytes") analyze_email('suspicious.eml')

Evidence Preservation Checklist

  • 1
    Export raw .eml file — right-click → Save As in most clients. Hash it immediately with SHA256 to prove it wasn't altered.
  • 2
    Screenshot headers — with timestamps visible. Include the full Authentication-Results field.
  • 3
    Document all IOCs — IP addresses, domains, URLs, attachment hashes. Use standard format (defang URLs).
  • 4
    Collect server logs — MTA (Postfix/Exchange) logs show exact receive timestamps. Preserve before log rotation.
  • 5
    Chain of custody — document who accessed evidence, when, and what actions were taken. Use write blockers if applicable.
  • 6
    Submit IOCs to threat intel — report phishing domains to APWG (eCrimeX), report IPs to AbuseIPDB, submit files to VirusTotal.
