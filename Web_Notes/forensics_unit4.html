Introduction to Cyber Forensics
What it is, why it matters, and how it works

What is Cyber Forensics?

Cyber Forensics (also called Digital Forensics or Computer Forensics) is the science of collecting, preserving, analyzing, and presenting digital evidence in a legally acceptable manner. Think of it like traditional police forensics — but instead of fingerprints and DNA, we deal with bytes, files, logs, and network packets.

🌍 Real-World Example Suppose an employee at a company is suspected of leaking confidential data. A cyber forensic investigator is called in. They take a forensic copy of the employee's laptop hard drive (without changing anything), examine browser history, USB connection logs, email outbox, and file access timestamps — and find evidence the employee copied files to a USB drive at 11:47 PM. This evidence is then presented in court.

Why is Cyber Forensics Important?

⚖️
Legal Evidence
Digital evidence (emails, logs, files) can be presented in court to prove or disprove a crime.
🔍
Incident Response
After a hack or breach, forensics tells you WHAT happened, WHO did it, and HOW it happened.
🛡️
Data Recovery
Recover accidentally or intentionally deleted evidence that criminals thought was gone forever.
📋
Compliance
Helps organizations meet legal and regulatory requirements by auditing systems.
🔐
Internal Investigations
Used by HR/legal teams to investigate fraud, IP theft, harassment, or policy violations.
🌐
Cybercrime Investigation
Track hackers, ransomware operators, fraudsters, and other cybercriminals across digital trails.

The Forensic Process (6 Phases)

1
Identification
Find all sources of evidence (laptop, phone, USB, cloud)
2
Preservation
Freeze evidence — prevent any change or corruption
3
Collection
Make forensic bit-for-bit copy (image) of the device
4
Examination
Extract relevant data from the image
5
Analysis
Interpret findings, build timeline, find IOCs
6
Reporting
Document everything in court-ready report

Key Principles

💡 Golden Rules of Cyber Forensics 1. Never work on original evidence. Always work on a forensic copy (image).
2. Preserve Chain of Custody (CoC). Document every person who touched the evidence.
3. Hash everything. MD5/SHA256 hashes prove the evidence was not modified.
4. Document everything. Every action must be recorded with timestamp and reason.
5. Be methodical. Follow a repeatable, documented process.

Q1. A company notices that confidential pricing data has been appearing on competitor websites. The IT team suspects an insider threat. List the steps a cyber forensic investigator should follow, and explain what type of forensics is most relevant in this case.

Q2. Why should a forensic investigator never work directly on original evidence? What could go wrong?
✓ Solution
A1. Steps:
1. Identification — Identify all devices the suspect used: laptop, USB drives, company email, cloud storage.
2. Preservation — Immediately suspend the account and take the laptop without powering it off (preserve RAM).
3. Collection — Create a forensic image of the laptop using FTK Imager or dd. Calculate MD5/SHA256 hash.

— Data corruption if tools crash during analysis
— Legal inadmissibility — courts can reject evidence that was not preserved correctly
— Destroying chain of custody — the integrity of the evidence is questioned

Storage Fundamentals
How data is physically and logically stored on devices

How Storage Works — The Big Picture

Before recovering any data, a forensic investigator must understand how storage devices work. Data doesn't just float in a hard drive — it is organized in a strict hierarchy: Disk → Partition → File System → Clusters → Sectors → Bytes.

Storage Device Types

DeviceFull NameHow it storesForensic Relevance
HDD Hard Disk Drive Magnetic platters + spinning disk. Data written via read/write head. Deleted data recoverable for months/years. Slack space retains old data.
SSD Solid State Drive NAND flash chips. TRIM command erases blocks automatically. TRIM makes recovery harder. Wear-leveling may scatter data across chips.
USB Flash USB Drive / Thumb Drive Flash memory (similar to SSD). FAT32 file system usually. Very common in insider threat cases. Can be cloned with dd.
SD Card Secure Digital Card Flash memory. Used in cameras, phones. Photos, videos, GPS data. FAT32/exFAT. Easy to image.
NVMe Non-Volatile Memory Express Ultra-fast SSD protocol connected via PCIe. Same challenges as SSD. Needs special adapters to image.
Optical CD/DVD/Blu-ray Laser burns pits into disc surface. Write-once discs preserve evidence reliably. Data can't be modified.

Key Storage Concepts

⚙️
Sector
The smallest unit of storage on a disk. Usually 512 bytes (old) or 4096 bytes (modern, 4K sectors). Every read/write happens in sector-sized chunks.
🔗
Cluster / Allocation Unit
A group of sectors. The file system works with clusters, not individual sectors. A typical cluster = 4KB to 64KB. Files occupy whole clusters, even if smaller.
🗂️
Partition
A logical division of a physical disk. One physical disk can have multiple partitions (C:, D:, etc.). Defined in MBR or GPT.
📍
MBR (Master Boot Record)
First 512 bytes of the disk. Contains the partition table and bootloader code. Old standard — supports max 4 partitions and 2TB disks.
📋
GPT (GUID Partition Table)
Modern replacement for MBR. Supports 128 partitions, disks larger than 2TB. Used on all modern UEFI systems.
🌑
Slack Space
Unused space at the end of a cluster. If a file is 1.5KB but the cluster is 4KB, the remaining 2.5KB is slack space — it may contain old data from previous files!
🔓
Unallocated Space
Disk space not assigned to any active file. Deleted files' data lives here until overwritten. Primary recovery target in forensics.
🔢
LBA (Logical Block Addressing)
Modern way to address sectors. Every sector gets a unique number starting from 0. Replaces the old CHS (Cylinder-Head-Sector) addressing.

Forensic Imaging — Making a Bit-for-Bit Copy

Before examining any disk, forensics requires creating an exact forensic image — a byte-for-byte copy of the entire disk including empty/unallocated space. This is not like copy-pasting files; it copies everything, including deleted files, slack space, and metadata.

BASH — Forensic Imaging with dd 
# STEP 1: Identify the target drive (do NOT mount it)
lsblk
# Output shows: sda (target suspect drive), sdb (forensic destination drive)

# STEP 2: Write-block the source (read-only access)
# On Linux use a hardware write blocker, or:
sudo blockdev --setro /dev/sda

# STEP 3: Create forensic image using dd
sudo dd if=/dev/sda of=/mnt/evidence/suspect.img bs=512 conv=noerror,sync status=progress
#  if  = input file (source — suspect drive)
#  of  = output file (destination — forensic image)
#  bs  = block size (512 bytes per sector)
#  conv=noerror,sync = continue on read errors, pad with zeros

# STEP 4: Verify image integrity with hash
md5sum /dev/sda
a3f4b9c1d2e5f8a7b6c4d3e2f1a9b8c7  /dev/sda
md5sum /mnt/evidence/suspect.img
a3f4b9c1d2e5f8a7b6c4d3e2f1a9b8c7  suspect.img
# Hashes MUST match — proves image is identical to original

# STEP 5: Better option — use dcfldd (forensic dd)
sudo dcfldd if=/dev/sda of=/mnt/evidence/suspect.img hash=md5,sha256 hashlog=/mnt/evidence/hash.log
⚠️ Write Blocker — Critical! Always use a hardware write blocker (e.g., Tableau T35u, WiebeTech) or a software write block before connecting a suspect drive. This prevents ANY write to the original disk — even the OS mounting the drive can alter metadata and timestamps!
📝 Task 2.1 — Storage Fundamentals Unit IV / Ch.2
Q1. A 10KB text file is stored on a hard drive with a cluster size of 32KB. How much slack space will this file create? What might the slack space contain?

Q2. You are given a suspect's USB drive. Write the exact dd command to create a forensic image of /dev/sdb and save it as /evidence/usb.img, then verify it with SHA256.
✓ Solution
A1. Slack Space Calculation:
File size = 10KB. Cluster size = 32KB.
The OS allocates the entire cluster = 32KB.
Slack space = 32KB − 10KB = 22KB
The 22KB of slack space may contain: fragments of previously deleted files, old versions of other documents, parts of emails, passwords, or any data that previously occupied those sectors on the disk. This is extremely useful in forensic investigations.

A2. dd Command:
sudo dd if=/dev/sdb of=/evidence/usb.img bs=512 conv=noerror,sync status=progress

Then verify with SHA256:
sha256sum /dev/sdb > /evidence/original_hash.txt
sha256sum /evidence/usb.img > /evidence/image_hash.txt
diff /evidence/original_hash.txt /evidence/image_hash.txt
If diff shows no output, the hashes are identical — image is verified.

File System Concepts
How the OS organizes and tracks files on disk

What is a File System?

A file system is the method an operating system uses to organize, store, and retrieve data on a storage device. Without a file system, a disk is just a random sequence of bytes — the OS wouldn't know where one file ends and another begins.

Think of a disk as a massive library. The file system is the catalog system — it records which shelf (cluster) each book (file) is on, how big it is, who owns it, and when it was last accessed.

Common File Systems

File SystemOSMax File SizeMax VolumeKey Forensic Features
FAT32 Windows, USB drives, cameras 4GB 8TB Simple structure. Deleted entries marked with 0xE5. Easy recovery.
NTFS Windows (Vista+) 16TB 256TB MFT records all files. ADS (Alternate Data Streams) can hide data. Journal logs changes.
exFAT Windows, SD cards, USB 16EB 128PB Used on large USB/SD cards. Better than FAT32 for large files.
ext4 Linux 16TB 1EB Journaling. Inode-based. Directory entries cleared on delete but inode+data may remain.
APFS macOS (2017+), iOS 8EB 8EB Copy-on-write. Snapshots. Clones. Encryption by default. Challenging to recover.
HFS+ macOS (older) 8EB 8EB Catalog file tracks all files. Timestamps stored as Mac epoch time.

Deep Dive: NTFS (Most Common in Windows Forensics)

NTFS is the most forensically rich file system. It uses a special database called the Master File Table (MFT) to track every file on the volume. Even deleted files leave behind MFT entries that can be analyzed.

📊
$MFT (Master File Table)
A database with one record (1KB each) for every file and folder. Contains filename, timestamps, permissions, and data location. First 26 entries are system files (e.g., $MFTMirr, $LogFile, $Bitmap).
📝
$LogFile (Journal)
NTFS transaction log. Records metadata changes before they happen (write-ahead logging). Can reveal recent file operations even if files are deleted.
🔄
$UsnJrnl (Change Journal)
Logs every file change (create, delete, rename, modify) with timestamps. Goldmine for forensics — records operations even after the file is deleted.
🕵️
ADS (Alternate Data Streams)
Hidden streams attached to a file. A malicious file can be hidden inside a legit file using ADS: notepad.exe:malware.exe. The main file appears normal in Explorer.
🕐
MACB Timestamps
Modified, Accessed, Changed (MFT modified), Born (created). NTFS stores two sets — in the file and in the MFT. Discrepancies reveal timestomping.
🗺️
$Bitmap
Tracks which clusters are allocated (used) and which are free. Each bit = one cluster. If bit is 0 = free = potentially recoverable. Critical for carving!

FAT32 File System — How Deletion Works

In FAT32, the file system uses a File Allocation Table (FAT) — essentially a chain-link map showing which clusters belong to each file. When you delete a file in FAT32:

Directory Entry Modified
The first character of the filename in the directory entry is replaced with 0xE5 (a special marker meaning "deleted"). The rest of the filename and metadata remains intact.
FAT Chain Zeroed Out
The FAT entries pointing to the file's clusters are set to 0 (free). But the actual data in those clusters is NOT erased — it's just marked as available for reuse.
Data Remains on Disk
Until the OS writes new data to those clusters, the original file data is physically present on disk. Recovery tools can read those clusters directly.
Recovery Window
The sooner you try to recover, the better. Every new file written to the drive risks overwriting the deleted file's clusters permanently.
🌍 Real Example — ADS Hidden Files An attacker hides a backdoor script inside an innocent text file using ADS:
type malware.exe > innocent.txt:hidden.exe
Windows Explorer shows innocent.txt as a tiny text file (0 bytes visible). The malware is hidden in the :hidden.exe stream. Forensic tools like streams.exe or dir /r can detect this.

📝 Task 3.1 — File System Analysis Unit IV / Ch.3
Q1. You run fls on a disk image and see the entry: r/r * 45-128-1: confidential_report.docx. What does the * mean? How would you recover this file using TSK?

Q2. A file is 3.8KB and the cluster size is 8KB. How much file slack exists? What important forensic information might exist in the slack space?

Q3. Explain the difference between $LogFile and $UsnJrnl in NTFS from a forensics perspective.
✓ Solution
A1. The * prefix in fls output means the file has been deleted. The inode number is 45.
Recovery command:
icat -o [offset] suspect.img 45 > confidential_report.docx
This reads the file's clusters directly using the inode and writes the raw bytes to a new file.

A2. Slack Space:
File size = 3.8KB, Cluster = 8KB
Slack space = 8KB − 3.8KB = 4.2KB
The slack space may contain: fragments of a previously stored password document, chat messages, parts of another file that occupied those sectors before this file was written there. This is called file slack and is regularly found to contain sensitive data.

A3. $LogFile vs $UsnJrnl:
$LogFile is a transaction journal used for crash recovery — it logs pending metadata changes so NTFS can roll back if the system crashes mid-write. It is circular (older entries overwritten) and typically holds only a few MB of recent operations.
$UsnJrnl is the change journal designed for long-term auditing. It records every file create/delete/rename/modify with timestamps and reason codes. It grows much larger and is invaluable in forensics because it shows file activity history even after files are deleted. Tool: usn.py, MFTECmd.

Windows Forensic Artifacts — Where Evidence Hides

Windows is full of artifacts — the OS records user activity in dozens of locations. A forensic investigator knows exactly where to look.

🗝️
Windows Registry
Central database for OS + app settings. Hives: SAM (user accounts), SYSTEM (hardware/services), SOFTWARE (installed apps), NTUSER.DAT (per-user settings), SECURITY. Contains USB history, recent docs, run keys (autorun malware), user accounts.
📜
Event Logs (.evtx)
Windows logs every login, process creation, service install, etc. Key IDs: 4624 (login success), 4625 (failed login), 4688 (process created), 7045 (service installed). Located in %SystemRoot%\System32\winevt\Logs.
⚡
Prefetch Files
Windows pre-loads frequently used programs. Prefetch file = proof a program was executed, includes timestamps of last 8 runs, and files accessed. Location: C:\Windows\Prefetch\. Format: PROGRAMNAME-HASH.pf
📎
LNK Files (Shortcuts)
Windows creates .lnk shortcut files every time you open a file. Contains: original file path, MAC timestamps, volume serial number, network share info. Even if original file is deleted, LNK file proves it existed!
🖱️
Jump Lists
Recent files for each application. Found in %AppData%\Microsoft\Windows\Recent\AutomaticDestinations. Proves files were opened even if deleted from disk.
🔌
USB Registry Keys
Windows records every USB device connected. Registry hive: SYSTEM\CurrentControlSet\Enum\USBSTOR. Shows device serial number, friendly name, first/last connected timestamps. Critical in insider threat cases!
🌐
Browser History
Chrome: %AppData%\Google\Chrome\User Data\Default\History (SQLite). Firefox: %AppData%\Mozilla\Firefox\Profiles\. Edge: %AppData%\Microsoft\Edge\User Data\. Contains URLs, visit times, download history.
🗑️
Recycle Bin ($I + $R)
Even after emptying Recycle Bin, the $I file retains metadata: original path, deletion time, file size. The $R file has the actual content. Located in C:\$Recycle.Bin\{SID}\.

📝 Task 4.1 — OS Artifacts & Terminology Unit IV / Ch.4

Data Recovery — Basics
Fundamentals of how and why data can be recovered after deletion

Why Can Deleted Data Be Recovered?

When you delete a file, the operating system does not actually erase the data. It merely removes the "pointer" to the data and marks the space as available for future use. Think of it like removing a book from a library catalog — the book is still on the shelf, you just can't find it through the catalog anymore.

This is the foundation of data recovery. The data remains physically present on the disk until new data is written on top of it (overwritten). The recovery window depends on: how much the disk is used after deletion — a fresh disk with no new writes may have near 100% recovery; a heavily used disk may have 0%.

What Happens When You Delete a File? (OS by OS)

🪟
Windows (NTFS)
1. File moved to Recycle Bin ($Recycle.Bin)
2. If Bin emptied: MFT record marked inactive, clusters marked free in $Bitmap
3. Data clusters NOT zeroed — still physically present
4. $LogFile and $UsnJrnl retain a record of the deletion event
🐧
Linux (ext4)
1. Directory entry removed (inode reference decremented)
2. Inode marked as free in inode bitmap
3. Block pointers in inode may be zeroed (ext4 does this!)
4. Data blocks marked free but content remains until overwritten
5. Note: ext4 is harder to recover from than ext3 due to zeroing
💾
FAT32 / exFAT
1. First byte of directory entry set to 0xE5
2. FAT chain entries zeroed (clusters marked free)
3. Actual cluster data NOT modified
4. Easiest to recover — filename and metadata partially preserved
⚡
SSD (TRIM enabled)
1. OS sends TRIM command to SSD controller
2. SSD controller physically erases deleted blocks proactively
3. Very hard to recover data from TRIM-enabled SSDs
4. Some enterprise SSDs log TRIM activity

File Carving — Recovery Without Metadata

When the file system metadata (MFT entries, directory entries, FAT chains) is destroyed but the actual file data still exists on disk, forensic investigators use file carving. Carving works by scanning raw disk sectors for known file signature bytes (magic numbers) at the start and end of files.

Q3. Why is it much harder to recover files from an SSD than from an HDD?
✓ Solution
A1. File Carving:
File carving is the process of extracting files from a disk image by searching for known file signature bytes (magic numbers) rather than relying on file system metadata. You use carving when:
— The file system is damaged or corrupted
— The partition table has been destroyed
— The disk was formatted (wiping metadata)
— File system metadata (MFT, FAT table) was partially overwritten
— You need to recover files from unallocated space with no metadata

A2. Foremost Command:
foremost -t jpg,pdf,zip -i phone_backup.img -o /evidence/carved/ -v

A3. SSD vs HDD Recovery Difficulty:
SSDs use the TRIM command — when a file is deleted, the OS immediately notifies the SSD controller, which proactively erases those NAND flash blocks. This happens at the hardware level, independent of the file system. By the time a forensic investigator images the SSD, the data blocks may already be physically zeroed. Also, wear leveling on SSDs constantly moves data across chips, making carving less reliable since a file's data may be split across multiple non-contiguous blocks with no predictable pattern.

Methods, tools, and techniques to recover deleted files from live and forensic images

Types of Deletion — What the User Thinks vs Reality

User ActionWhat User ThinksWhat Actually HappensRecovery Possible?

Empty Recycle Bin "File is permanently gone" MFT entry marked inactive, clusters marked free. Data remains. ✅ High — with Recuva, Autopsy, TSK

Step-by-Step: Deleted File Recovery Methodology

Stop Using the Drive Immediately
Every write operation risks overwriting the deleted file's clusters. Power off the device or use a hardware write blocker. The longer the drive is used, the lower the recovery chance.
Create a Forensic Image
Use dd, dcfldd, or FTK Imager to create a bit-for-bit copy. Always work on the image, never on the original. Calculate MD5/SHA256 hash to document integrity.
Check the Recycle Bin / Trash
If files were not emptied from the Recycle Bin, they can be restored directly. Examine C:\$Recycle.Bin\{UserSID}\$I* files for metadata of deleted items.
File System-Based Recovery (via MFT/FAT)
Use Autopsy, Recuva, or TSK's fls + icat to find and recover files whose MFT entries or FAT directory entries are still intact but marked as deleted.
Carve Unallocated Space
If file system metadata is gone, run Foremost/Scalpel/PhotoRec on the raw image to carve files by signature from unallocated space.
Examine Slack Space & Fragments
Bulk Extractor and Strings can extract fragments of text, emails, URLs from slack space even if complete file recovery is impossible.
Verify & Document Recovered Files
Hash each recovered file. Verify it opens correctly and is not corrupted. Document recovery method, timestamp, and source sector location for the court report.

Formatted Partition Recovery
Recovering data after formatting, repartitioning, and volume deletion

What is Formatting?

Formatting prepares a disk or partition for use by creating a new file system. Users often format a drive to "wipe everything" before selling, returning, or destroying a device. However, formatting is not the same as erasing.

⚡
Quick Format
Creates a new, empty file system metadata structure (new MFT or FAT tables). Does NOT touch data clusters. Old data remains 100% intact. Recovery is highly successful. This is what most users do.
🔄
Full Format
On Windows Vista+, a full format writes zeros to every sector. Data is overwritten once. Recovery with standard tools = very difficult. Some advanced hardware techniques may find remnants.
✂️
Repartitioning
Deleting and recreating partitions destroys the partition table entries. Old data often survives in the same location, recoverable by reconstruction of the original partition layout.
🔃
File System Change
Reformatting from NTFS to FAT32 (or vice versa) creates a completely different metadata structure, but again — data clusters are not overwritten. Carving tools can still recover files.

What Formatting Actually Destroys vs Preserves

ComponentQuick FormatFull Format
MBR / Partition Table⚠️ Recreated⚠️ Recreated
File System Metadata (MFT/FAT)⚠️ Replaced with new, empty version⚠️ Replaced + zeroed
Volume Boot Record (VBR)⚠️ Overwritten⚠️ Overwritten + zeroed
Actual File Data (clusters)✅ UNTOUCHED — fully recoverable❌ Overwritten with zeros
Slack Space✅ Preserved❌ Zeroed
Free/Unallocated Space✅ Unchanged❌ Zeroed
File Names❌ Lost (new empty MFT)❌ Lost
Directory Structure❌ Lost❌ Lost

Recovery from a Quick-Formatted Drive

After a Quick Format, all file data is still present. The only things destroyed are the file system metadata tables. Recovery strategy:

Image the Formatted Drive
Use dd/dcfldd to capture every bit including the now-empty file system structure and intact data clusters. Use write blocker. Calculate hashes.
Attempt File System Reconstruction
TestDisk can detect and rebuild the original partition table and file system structures by scanning for signature bytes. This often recovers filenames and directory structure too!
Carve Directly from Raw Image
Even if TestDisk fails, Foremost/Scalpel/PhotoRec can extract files by signatures from the raw data — without needing any file system metadata.
Extract Strings and Fragments
Use Bulk Extractor and strings to extract email addresses, URLs, partial text documents, and other structured data that may not carve as complete files.

🎯 Master Task — Full Unit IV Scenario Exam Level
Scenario: Cyber Crime Cell receives a complaint against Mr. Raj, a senior accountant at FinCorp Ltd. He is suspected of stealing client financial data and selling it to a competitor. When investigators arrive, Mr. Raj is still at his workstation. His Windows 10 PC is running, and a 32GB USB drive (Sandisk, NTFS, Quick Formatted 2 hours ago) is in his drawer.

✓ Model Answer
Q1 — Initial Response (6 marks):
Do NOT power off — capture RAM first. The PC is live and running. RAM contains volatile evidence that disappears on power off: running processes, network connections, encryption keys, clipboard contents, logged-in sessions, and potentially malware hiding only in memory.

Step 1 — RAM Acquisition: Use WinPmem or Magnet RAM Capture on a write-protected USB to dump RAM to a file. Hash the dump immediately.
Step 2 — Document the scene: Photograph the screen, connected devices, and workstation layout before touching anything.
Step 3 — Network isolation: Disconnect ethernet, disable WiFi (without powering off) to prevent remote wiping.
Step 4 — Collect the USB drive — place in anti-static evidence bag with CoC label.
Step 5 — Graceful shutdown (if warranted) or pull-plug depending on jurisdiction protocol.

Evidence sources: PC hard drive, USB drive (Sandisk), RAM dump, network router logs, email server logs, company SIEM, CCTV footage of workstation.

Q2 — Forensic Imaging (8 marks):
For both drives, use a hardware write blocker (Tableau T35u) first.
PC Hard Drive (/dev/sda):
dcfldd if=/dev/sda of=/evidence/raj_pc.img hash=md5,sha256 hashlog=/evidence/pc_hashes.txt bs=512 conv=noerror,sync status=progress

USB Drive (/dev/sdb):
dcfldd if=/dev/sdb of=/evidence/raj_usb.img hash=md5,sha256 hashlog=/evidence/usb_hashes.txt bs=512 conv=noerror,sync status=progress

Integrity Verification:
sha256sum /dev/sda > /evidence/orig_hash_pc.txt
sha256sum /evidence/raj_pc.img > /evidence/img_hash_pc.txt
diff /evidence/orig_hash_pc.txt /evidence/img_hash_pc.txt (must show no difference)
Seal original drives in tamper-evident evidence bags with signatures, date, and hash values recorded on the bag and in the CoC form.

Q3 — Quick Format Not Truly Empty (6 marks):
A Quick Format only rewrites the file system metadata — specifically the $MFT (Master File Table) and the $Bitmap are reset to empty. The data clusters are never touched or zeroed. Every file that Mr. Raj had on the USB still occupies the exact same sectors on the drive — the OS simply no longer has a map to find them. It's like tearing out the index of a book: the content is still there, you just can't look it up anymore.

Recovery approach:
1. TestDisk — Can find the old NTFS partition structure and may restore original file names: testdisk raj_usb.img → Analyse → Quick Search
2. PhotoRec — Carves 480+ file types from the raw data: photorec raj_usb.img → Whole disk → Other
3. Foremost/Scalpel — Additional carving: foremost -t all -i raj_usb.img -o /evidence/carved/

✓ Short Answers
1. Chain of Custody: A chronological record documenting who collected, handled, examined, or transferred evidence. Critical in court because it proves the evidence was not tampered with — if CoC is broken, defense can argue evidence was planted or altered, rendering it inadmissible.

2. Volatile vs Non-Volatile:
Volatile (lost on power off): RAM contents, running processes, network connections, clipboard, logged-in users.
Non-volatile (persists): Hard drive files, registry, installed programs, USB connection history, event logs.

3. FF D8 FF: This is the JPEG file signature (magic number/header). In file carving, tools like Foremost scan raw disk sectors looking for this byte sequence. When found, the carver knows a JPEG file starts here and carves bytes until the JPEG footer FF D9 is found, extracting a complete image even with no file system metadata.

4. MACB: Modified-Accessed-Changed (MFT)-Born. If M time is earlier than B (Born/Created) time, this is impossible for a legitimately created file and indicates timestomping — the suspect deliberately altered the Modified timestamp to make the file appear older than it is. This is a strong sign of anti-forensic activity.

5. SSD Recovery: SSDs use the TRIM command — when a file is deleted, the OS notifies the SSD to proactively erase those flash blocks for performance. By the time forensics begins, data may already be physically wiped at hardware level. Also, wear-leveling scrambles data across NAND chips unpredictably.

6. Slack Space: The unused area at the end of a cluster after a file ends but before the cluster boundary. A 1KB file in a 4KB cluster leaves 3KB of slack. Slack may contain fragments of previously stored files, passwords, text snippets, or email data from when those sectors were last used by a different file.

7. Quick vs Full Format: Quick Format: Only file system metadata (MFT/FAT tables) is reset — data clusters untouched, high recovery chance. Full Format (Vista+): Every sector written with zeros — data is overwritten, essentially irrecoverable with standard tools.

8. Hash in Forensics: A hash (MD5, SHA256) is a unique mathematical fingerprint of data. Same input always gives same output; any change in the data gives a completely different hash. In forensics, it proves: (1) the forensic image is an exact copy of the original (both should have identical hashes), and (2) evidence was not modified after collection. If hashes match before and after, no tampering occurred.
